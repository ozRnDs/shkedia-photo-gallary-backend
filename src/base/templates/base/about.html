About Project Shkedia

Every day, we accumulate massive amounts of MEDIA on our phones. Taking pictures and videos is easy, so we often try 10 times until we get the best photo. If that's not enough, the MEDIA we create gets mixed up with spam from different apps. After all that, we need to save copies of the important MEDIA, a never-ending process of screening, copying, backing up, and more. At the end of the day, the entire MEDIA usually sits on a hard drive that is rarely opened.

The idea of Project Shkedia was born as a response to the challenges outlined above. It aims to enable effortless and secure management of our MEDIA through the following features: seamless uploading of MEDIA to the cloud, efficient cleaning of backed-up MEDIA from the phone, combining all family MEDIA, screening for spam, and suggesting deletions. Additionally, it will organize MEDIA into albums based on events and people and extract INFORMATION for our benefit.

Behind The Scenes

{% comment %} {% endcomment %}

The system is built for a cloud environment with horizontal scaling in mind. Each component is designed as a self-sustained container, and communication between the components is based on RESTful API.

The internal micro-services were built using FastAPI as it is lightweight and provides a ton of flexibility to handle complex tasks. This layer enables quick database or storage changes without the need to dive into a huge code base. It serves the backend layer, supporting it with complex logic. None of these components have access to encrypted data; they act as a pipe to the rest of the system.

The core business logic is implemented in the Backend Layer, divided into the Upload Service and Gallery (Download) Service. The upload process is memory and time-consuming, and it handles the files, making it more prone to attacks. This component only has the public key of the system. Even if attacked or compromised, the rest of the system can remain safe and functional. The Gallery Backend was built using Django because of its strong template engine, handling presentation and searching logics in the system.

The frontend is also divided into two core clients: Android (Kotlin) for uploading media and cleaning the phone, and a web-based application to view and search the data. Most of the Web-UI is based on the template designed by TemplateMo (Catalog-Z Company).

DATA

{% comment %} {% endcomment %}

The system's data is divided into two categories: INFORMATION and MEDIA. MEDIA includes binary files like images and videos, while INFORMATION encompasses the rest, each with distinct properties and handling procedures.

MEDIA consumes significant storage space, and frequent access is not required. Hence, it will be stored in a dedicated storage system, possibly even a cold one.

In contrast, constant access is crucial for INFORMATION, forming the foundation for all system logics. However, for everyday use, a compact copy of MEDIA (thumbnail) is stored in the database.

AI (Future Features)

By using various open-source Deep Learning Models, this system can grow and align with the project's goals. Insights about the images will be gathered to enhance search. Automatic classification of images will empower albums creation. For example, YOLOv7 will extract the objects exist in the images. Using face detection and recognition models, personal albums can be created. With DeepFace, face attributes and emotions can also be extracted. All of these enhancements will contribute to a better search experience.
